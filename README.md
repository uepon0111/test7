# プロセカ リザルト管理

Project Sekai のリザルト画像を Google Drive に保存・一覧管理する GitHub Pages 用の静的Webアプリです。

## ファイル構成

機能・役割ごとにファイルを分割しています。`index.html` 1枚で完結するSPA構成です
(以前は「読み取り設定」が `settings.html` という別ページでしたが、ページ遷移すると
Googleログインの状態が失われる問題があったため、index.html内のモーダルに統合しました)。

```
index.html            唯一のHTML。リザルト一覧・アップロード/編集・読み取り設定・実測ログ、
                       すべてこの中のモーダルとして完結します。

css/
  base.css            全体レイアウト・フィルターUI・カードグリッド
  modals.css          画像モーダル・アップロード/編集モーダル
  settings.css        読み取り設定モーダル (範囲エディタ) 専用スタイル
  analysis.css        実測ログモーダル・警告表示・候補選択チップのスタイル

js/
  config.js           定数 (難易度定義・色・Driveフォルダ名・デフォルト読み取り範囲など)
  utils.js            汎用ヘルパー (文字列正規化・レーベンシュタイン距離・日付整形など)
  state.js            index.html 内で共有するグローバル状態
  device-profiles.js  機種プロファイルのCRUD (localStorage保存・自動選択・エクスポート/インポート)
  music-db.js         楽曲マスターDB取得・曲名のファジーマッチング・逆算候補検索
  result-resolver.js  OCR結果とDBを突き合わせて曲名・難易度・レベルを総合的に決定する判断ロジック
  drive-client.js     Google Drive API の低レベル汎用処理 (検索・作成・アップロード・更新・削除)
  drive-records.js    リザルトデータ⇔Driveファイルの変換・取得・保存 (新方式/旧方式)
  personal-best.js    自己ベストの判定・グルーピング処理
  ocr-analyzer.js     Tesseract.js によるOCR解析 (難易度・レベル・曲名・判定内訳・コンボ数)
  analysis-log.js     実測ログモーダル・警告/候補表示のUI
  auth.js             Googleログイン/ログアウト
```

## Google Drive 上のデータ構成

### 新方式 (このバージョンから)

```
プロセカリザルト/            (ルートフォルダ)
  └─ Results/               (新規アップロードは全てここにフラットに保存)
       ├─ MS29 曲名A (FC) 2026-07-08.png
       ├─ EX27 曲名B (FC-3) 2026-07-09.png
       └─ ...
```

曲ごとにサブフォルダを作らず、1つのフォルダに全リザルト画像を保存します。曲名・レベル・難易度・
判定内訳(PERFECT/GREAT/GOOD/BAD/MISS)・コンボ数などのデータは、各画像ファイルの
Drive カスタムプロパティ (`properties`) に保存しています。これにより、曲数が増えても
フォルダの作成・検索コストが増えず、一覧取得もフォルダ横断の突き合わせが不要になります。

### 旧方式 (読み取り専用の後方互換)

これまで運用していた `プロセカリザルト/FC/{レベル}{難易度} 曲名/FC` という構成のデータも、
引き続き一覧に表示されます。旧データを編集して保存すると、その時点で自動的に新方式
(Resultsフォルダ)へ移行されます(画像本体の再アップロードは不要です)。

## OCR読み取りの仕組み

### 読み取る項目

難易度・**レベル(新規追加)**・曲名・判定内訳(PERFECT/GREAT/GOOD/BAD/MISS)・コンボ数の
6項目を読み取ります。レベルは以前は曲名から逆引きするだけでしたが、画像から直接OCRで
読み取るようになり、他の項目と突き合わせて曲名の特定に使えるようになりました。

### 項目ごとの二値化

難易度バッジは難易度ごとに背景色が異なり(HARDは黄、MASTERは紫など)、固定のしきい値では
明るい背景色と白文字の区別がつかなくなることがありました。そこで、切り出した画像ごとに
ヒストグラムから最適なしきい値を求める「大津の二値化」を使い、さらに「画素数が少ない方
(=文字)/多い方(=背景)」を都度自動判定することで、バッジの配色によらず白背景+黒文字に
統一しています。加えて、項目ごとに適切な拡大率・文字種のホワイトリスト・ページ分割モードを
Tesseract.jsに指定することで認識精度を上げています。

自動判定でうまくいかない機種がある場合は、設定モーダルの各範囲タブから二値化の向き
(自動 / 文字が濃い色 / 文字が薄い色)を手動で固定できます。実際にどう二値化されたかは
「実測ログを見る」から画像として確認できます。

### 曲名・難易度・レベルの総合判断

曲名・難易度・レベル・総ノーツ数(判定内訳の合計)を突き合わせ、自信の持てない項目を
他の項目から補う形で最終的な値を決定します(`result-resolver.js`)。

- 曲名の読み取りに自信が持てない場合、読み取った難易度・レベル・総ノーツ数から楽曲を逆算します
  (その難易度でその総ノーツ数に近い楽曲をマスターDBからリストアップし、最も近いものを採用)。
- 曲名は特定できても難易度の読み取りがあいまいな場合は、同じ楽曲の中で総ノーツ数が最も近い
  難易度に補正します。
- 特定した楽曲のDB上の総ノーツ数と、実際に読み取った判定内訳の合計が食い違う場合は警告を出します
  (判定内訳の読み取りミスの可能性を示すサインです)。
- コンボ数は総ノーツ数を超えることが物理的にありえないため、超えている場合は「コンボ数の誤読、
  または範囲に別の数値が写り込んでいる可能性がある」警告を出します。
- 自動推定した楽曲が間違っている場合は、編集フォームの候補チップ、または実測ログモーダルの
  候補一覧からワンクリックで選び直せます。

判定内訳・コンボ数はあくまで実測値として扱い、DBの値で勝手に上書きすることはありません。
DB側の値を正として採用するのは、楽曲マスターに紐づく固定情報である曲名・難易度・レベルのみです。

### 実測ログ

編集フォームの「実測ログを見る」から、画像1枚ごとに
- 項目ごとの切り出し座標(元画像上のピクセル位置)
- 実際にOCRへ渡した二値化後の画像
- OCRの生テキストと解釈結果・信頼度
- 楽曲の特定方法(曲名から/統計から逆算/特定できず)と警告・候補一覧

を確認できます。読み取り精度のチューニングや、誤読の原因調査に使ってください。

## 読み取り範囲の設定 (設定モーダル)

「読み取り設定」ボタンから開くモーダルで、難易度・レベル・曲名・判定内訳・コンボ数を
読み取る範囲を、スクリーンショットの機種(解像度・アスペクト比)ごとに保存できます。範囲は
ドラッグ&ドロップまたはピクセル単位の数値入力のどちらでも調整でき、双方向に反映されます。
以前は別ページ(`settings.html`)でしたが、ページ遷移のたびにGoogleログインが切れてしまう
問題があったため、index.html内のモーダルに統合しました。設定を開いてもページ遷移が発生しないため、
ログインが切れることはありません。

アップロード時は画像の解像度・アスペクト比から最も近い機種プロファイルが自動的に選ばれますが、
アップロード画面の「機種プロファイル」欄からいつでも手動で切り替えられます
(切り替えると、その場でその機種の読み取り範囲を使って再解析されます)。

機種プロファイルはブラウザの localStorage に保存されます(サーバーを持たない静的サイトのため)。
別のブラウザ・端末でも使う場合は、設定モーダルの「エクスポート/インポート」で設定ファイル(JSON)を
移行してください。以前のバージョンで保存したプロファイルを開いた場合、今回追加した「レベル」の
読み取り範囲は自動的にデフォルト値で補完されます(手動で調整してください)。

## コンボ数・レベルの読み取り範囲について

添付いただいたリザルト画面のサンプルを実測してデフォルト値を設定していますが、機種によって
UIレイアウトが異なる場合があります。実際の見え方に合わせて、設定モーダルから調整してください
(判定内訳の範囲も、PERFECT/GREATの表示位置に応じて微調整が必要な場合があります)。
